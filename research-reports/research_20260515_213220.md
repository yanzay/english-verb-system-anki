# Anki Preset Auto-Bind on `.apkg` Import — Root Cause & Path Forward

**Date:** 2026-05-15  
**Question:** Our genanki-built `.apkg` embeds a custom FSRS preset; on import Anki shows "Default". Other shared decks reportedly auto-bind their preset. Why, and what's the proper fix?

---

## Executive summary

The mechanism is now fully understood. Anki's `.apkg` importer has a single boolean flag — `with_deck_configs` — that controls whether bundled deck configs survive import. **The flag's default is `false`.** When `false`, the importer's `gather.rs` calls `reset_decks()` which forcibly **rewrites every imported deck's `config_id` back to `1` (the user's Default preset)**. Our hand-injected dconf JSON is technically present after import, but no deck references it, so it's invisible.

Decks that *do* auto-bind their preset (AnKing, Migaku, etc. when working) achieve this by being **exported with `with_deck_configs = true`** at *export* time. genanki cannot do this — it builds its own SQLite from scratch and never invokes Anki's official export pipeline.

There are three real fixes, in increasing order of effort and reward:
1. **Ship the standalone `english_verb_system_preset.json`** with one-time apply instructions (already done — works on every Anki ≥ 23.10).
2. **Ship a tiny AnkiConnect-driven `apply_preset.py`** for a one-line zero-click apply.
3. **Migrate the build pipeline from genanki → official `anki` PyPI package**, which calls Anki's real export with `with_deck_configs=true` and produces an `.apkg` that auto-binds on import. ~4–8 hours of migration work; permanent fix.

Recommendation: do (3). It permanently retires several other workarounds (`embed_fsrs_preset` SQLite hack, model-id management, preset JSON file) and gives us first-class access to FSRS parameters, modern Anki schema (v18 / collection.anki21b), and the `meta` protobuf metadata.

## Theme 1 · The exact mechanism (smoking gun)

Anki's Rust importer (`rslib/src/import_export/package/apkg/`) processes a `.apkg` in three stages:

```
                    ┌───────────────────────────────────────────────┐
ImportAnkiPackage   │ apkg/import/mod.rs:62                         │
    Options         │   with_deck_configs: bool (default: FALSE)    │
                    └─────────────────┬─────────────────────────────┘
                                      │
                                      ▼
                    ┌───────────────────────────────────────────────┐
                    │ gather.rs:61-64                               │
                    │   if !with_deck_configs:                      │
                    │       reset_config_ids = true                 │
                    └─────────────────┬─────────────────────────────┘
                                      │
                                      ▼
                    ┌───────────────────────────────────────────────┐
                    │ gather.rs:117-118  ← THIS IS THE BLOCKER     │
                    │   if reset_config_ids:                        │
                    │       deck.config_id = 1   // ← reset to def. │
                    └─────────────────┬─────────────────────────────┘
                                      │
                                      ▼
                    ┌───────────────────────────────────────────────┐
                    │ apkg/import/decks.rs:188-189                  │
                    │   if other.config_id != 1 {                   │
                    │       normal.config_id = other.config_id;     │
                    │   }                                           │
                    └───────────────────────────────────────────────┘
```

**Two independent gates** must both pass for a bundled preset to bind on import:

1. The package was **exported** with `with_deck_configs=true` (so dconfs are *present* and config_ids are *not reset*).
2. After step 1, each deck's `config_id != 1` (so the binding line `decks.rs:188-189` actually fires).

Our package satisfies gate 2 but not gate 1: gate 1 is a property of how the package was *created* via Anki's official exporter, which genanki bypasses.

### Where we already fail
genanki produces `collection.anki2` (legacy schema v11) and never communicates an `with_deck_configs` intent to Anki. On import, Anki defaults to `with_deck_configs=false`, runs `reset_decks()`, our deck `config_id`s become `1`, and `decks.rs:188-189` falls through.

This is **by design**: from the Anki maintainer (issue #3023), shared decks should not be allowed to alter the user's defaults silently.

## Theme 2 · Why every "easy fix" fails

Subagent 3 proposed adding a `meta` file inside the zip with `{'ver': 1}`. **This is a false lead.** The `meta` file exists in `.colpkg` (collection packages), not user-imported `.apkg` files, and even there it only carries the schema version and zstd flag — it has no field that controls `with_deck_configs`. Adding a `meta` file to our `.apkg` will be silently ignored by Anki.

Other false leads we should not pursue:
- ❌ Setting magic values in `col.conf` (no such mechanism)
- ❌ Reserved `config_id` ranges (Anki treats only `1` specially)
- ❌ Using `.colpkg` instead — `.colpkg` overwrites the user's entire collection on import, which is unacceptable
- ❌ Patching genanki to set a flag in the SQLite — there is no flag stored in the SQLite; the gate is on the *importer's* options dict

## Theme 3 · How AnKing / Migaku actually do it

When a popular deck *does* auto-bind on import, it's not because the maintainer found a clever SQLite trick. It's because they used Anki's official export path with `with_deck_configs=true` enabled. Two practical realisations:

1. **Manual export from Anki Desktop**: File → Export → check "Include deck options". This sets the flag; the resulting `.apkg` has `config_id != 1` for the bound deck and `dconf` populated. On import, Anki sees a non-default config_id, looks up the dconf, finds it, and binds it.
2. **Programmatic export via the official `anki` Python package**: `Collection.export_anki_package(out_path, options=ExportAnkiPackageOptions(with_deck_configs=True), limit=…)`. This is the same code path as the GUI export.

When AnKing users complain that their preset "resets" anyway, it's because their Anki version (< 24.04) had bugs in the same flag handling, or because the deck was originally set to dcid=1 at export time (the secondary `dcid==1` skip).

### Key takeaway
There's no shortcut. To get auto-binding we **must** invoke Anki's official export path — which means moving off genanki.

## Theme 4 · genanki vs the official `anki` PyPI library

| Capability | genanki 0.13 | official `anki` 25.9.4 |
|---|---|---|
| Notes / cards / models | ✓ | ✓ |
| Deck-options preset binding on import | ✗ (must SQLite-hack) | ✓ native, exports with `with_deck_configs=true` |
| FSRS parameters first-class | ✗ | ✓ |
| Modern schema (v18, collection.anki21b) | ✗ stuck on v11 | ✓ |
| `meta` protobuf metadata | ✗ | ✓ auto |
| Active maintenance | ✗ inactive since Oct 2021 | ✓ tracks Anki releases |
| Install footprint | tiny | ~50 MB (protobuf, requests, orjson, no Qt) |
| Migration cost | — | ~4–8 hours |

The official package is a Python wrapper around the Rust `anki` library that powers Anki Desktop. It has no GUI dependency. There's a known one-time hiccup with circular imports during initial use, easily worked around. Once the migration lands:
- `embed_fsrs_preset()` (~110 lines of SQLite hackery) is **deleted**
- `validate_apkg.py` model-id collision check stays (still useful)
- `english_verb_system_preset.json` becomes optional belt-and-braces (or deleted)
- We get FSRS parameter exposure (we can train per-user weights and ship them)
- We can produce `.colpkg` for power users wanting a full snapshot
- We can move to the modern compressed `collection.anki21b` schema

## Recommendation

**Do all three, in order:**

| Phase | Effort | Effect |
|---|---|---|
| **Now** (already shipped) | 0 | Standalone `english_verb_system_preset.json` + one-time apply docs. Works on every Anki ≥ 23.10. |
| **+1 hour** | 1 hr | `apply_preset.py` AnkiConnect helper — user runs `python apply_preset.py` after import, it binds the preset to all 52 sub-decks via Anki's local HTTP API. Zero clicks for users with AnkiConnect. |
| **+4–8 hours** | 4-8 hr | **Migrate `build_anki_package.py` to the official `anki` package.** Calls the real exporter with `with_deck_configs=true`. Removes 3 workarounds permanently. Auto-binds on every Anki ≥ 23.10. |

I'd skip the AnkiConnect step and go straight to the migration — it makes the apply-script unnecessary and is the proper long-term solution.

## Summary bullets

- **Root cause**: Anki's importer defaults `with_deck_configs=false`, which silently resets every imported deck's `config_id` to `1`. Our preset JSON is then unreferenced by anything.
- **Why other decks work**: They're exported with `with_deck_configs=true` (a property of the exporter, not the SQLite). genanki bypasses Anki's exporter entirely so it can't set this flag.
- **The "meta file" idea is a false lead** — that file only exists in `.colpkg` and doesn't carry deck-config import settings.
- **Permanent fix**: Migrate from genanki to the official `anki` PyPI package (~4–8 hours). It uses Anki's real export with the right flag and produces an `.apkg` that auto-binds the preset on import on every modern Anki version.
- **Side benefits**: removes ~150 lines of SQLite-hacking, exposes FSRS parameters, modern schema, smaller `.apkg` (zstd), removes need for the standalone preset JSON.
